# Code Quality Standards

## Purpose

Defines quality standards for generated MCP server code including Python version requirements, uv dependency management, type hints, docstrings, validation, and error handling requirements.

## CRITICAL: Python and Dependency Management Requirements

**Python Version:** Python 3.10+ REQUIRED
- Use modern type hints (PEP 604 union syntax: `str | None` instead of `Optional[str]`)
- Use structural pattern matching where appropriate
- Target Python 3.10 minimum, 3.11+ preferred

**Dependency Management:** uv ONLY
- **NEVER install dependencies globally with pip**
- **ALL dependency management through uv:**
  - `uv sync` - Install dependencies from lock file
  - `uv add package` - Add new dependency
  - `uv run command` - Run commands in virtual environment
  - `uv pip` - Pip interface (when necessary for compatibility)

**Project Structure Requirements:**
- pyproject.toml as PRIMARY dependency specification
- uv.lock for reproducible builds (generated by uv)
- requirements.txt as FALLBACK ONLY (generated from uv: `uv pip compile pyproject.toml`)
- NO setup.py (deprecated in favor of pyproject.toml)

**pyproject.toml Structure:**
```toml
[project]
name = "mcp-server-name"
version = "0.1.0"
description = "MCP server description"
requires-python = ">=3.10"
dependencies = [
    "mcp>=0.1.0",
    "pydantic>=2.0.0",
    "httpx>=0.24.0",
    # ... other runtime dependencies
]

[project.optional-dependencies]
dev = [
    "black>=23.0.0",
    "ruff>=0.1.0",
    "mypy>=1.0.0"
]
test = [
    "pytest>=7.0.0",
    "pytest-cov>=4.0.0",
    "pytest-asyncio>=0.21.0"
]

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[tool.hatch.build.targets.wheel]
packages = ["src"]
```

## Import Standards (CRITICAL - Absolute Imports ONLY)

**MANDATORY RULE: ALL imports MUST use absolute imports starting from the package root**

**Pattern:** `from src.module.submodule import Item`
**NEVER:** `from .module import Item` or `from ..module import Item`

**Why Absolute Imports:**
- Eliminates import path ambiguity
- Prevents circular dependency issues
- Makes refactoring safer
- Enables better IDE support and navigation
- Required for proper pytest execution
- Compatible with all Python execution contexts

**Examples:**

✅ **CORRECT (Absolute Imports):**
```python
# In src/server.py
from src.config import Config
from src.tools.factory import ToolFactory
from src.utils.logging_utils import setup_logging

# In src/tools/chats/chat_list_tool.py
from src.tools.base import BaseTool
from src.models.chat import Chat
from src.utils.validation import ToolInputValidator
```

❌ **INCORRECT (Relative Imports - FORBIDDEN):**
```python
# In src/server.py
from .config import Config                    # WRONG - DO NOT USE
from .tools.factory import ToolFactory        # WRONG - DO NOT USE

# In src/tools/chats/chat_list_tool.py
from ..base import BaseTool                   # WRONG - DO NOT USE
from ...models.chat import Chat               # WRONG - DO NOT USE
from ...utils.validation import ToolInputValidator  # WRONG - DO NOT USE
```

**Enforcement:**
- All code generation MUST follow absolute import pattern
- Validation phase MUST check for relative imports (grep for `from \.` patterns)
- Any relative import found is a BLOCKING validation failure
- Package root is always `src/`
- All absolute imports start with `src.`

## Type Hints (100% Coverage Required)

All functions/methods MUST have complete type hints including parameters, return types, and exceptions.

**Required:** typing imports, full annotations
**Pattern:** Use `from typing import Dict, List, Any` and modern union syntax `str | None`

## Google-Style Docstrings (Required for Public Interfaces)

All public functions/methods/classes MUST have Google-style docstrings with Args, Returns, Raises sections.

## Pydantic Validation (All External Inputs)

All external inputs MUST use pydantic models with Field validators.

## Error Handling (No Internal Exposure)

Error messages must be user-friendly without exposing internals. Use custom exception hierarchy.

## Logging (Appropriate Levels)

INFO: Normal operations, WARNING: Recoverable issues, ERROR: Failures, DEBUG: Detailed debugging

## Code Formatting

Black formatting (line length 100), imports ordered, no unused imports, consistent naming (snake_case functions, PascalCase classes)

## MCP Tool Registration (CRITICAL)

The `list_tools()` handler MUST return `mcp.types.Tool` objects, NOT dictionaries.

**Correct Pattern:**
```python
from mcp.types import Tool

@mcp_server.list_tools()
async def list_tools() -> list[Tool]:
    """List all available MCP tools."""
    tools = factory.get_all_tools()
    return [
        Tool(
            name=t.get_definition()["name"],
            description=t.get_definition().get("description"),
            inputSchema=t.get_definition().get("inputSchema", {})
        )
        for t in tools
    ]
```

**WRONG Pattern (causes "'dict' object has no attribute 'name'" error):**
```python
@mcp_server.list_tools()
async def list_tools() -> list[dict]:  # WRONG return type
    return [tool.get_definition() for tool in tools]  # Returns dicts, not Tool objects
```

**Why This Matters:**
- The MCP SDK expects `Tool` objects with `.name`, `.description`, `.inputSchema` attributes
- Returning dictionaries causes `AttributeError: 'dict' object has no attribute 'name'`
- This error occurs when the MCP client tries to fetch the tool list

## Tool Response Formatting (CRITICAL)

Tool execution results must be properly JSON-serialized for MCP clients.

**Correct Pattern:**
```python
import json

@mcp_server.call_tool()
async def call_tool(name: str, arguments: dict) -> dict:
    result = await tool.execute(arguments)
    return {
        "content": [{
            "type": "text",
            "text": json.dumps(result, indent=2)  # Proper JSON formatting
        }]
    }
```

**WRONG Pattern (loses data structure):**
```python
return {
    "content": [{
        "type": "text",
        "text": str(result)  # WRONG: Python repr, not JSON
    }]
}
```

**Why This Matters:**
- `str(result)` produces Python repr format: `{'key': 'value'}` (single quotes)
- JSON requires double quotes: `{"key": "value"}`
- MCP clients expect parseable JSON, not Python string representation
